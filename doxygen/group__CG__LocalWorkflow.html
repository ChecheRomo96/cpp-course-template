Building the Project

This page explains how to configure, build, and run the project locally using CMake. It is part of the Local Workflow section of the course guide.

1.0 Configure the Build Directory

To maintain a clean repository structure, always use an out-of-source build. This keeps generated files separate from your source code.

Configure the project:

rm -rf build
cmake -B build -DCMAKE_BUILD_TYPE=Debug

This creates the build directory and generates all required build system files inside build/, leaving the repository root unmodified.

1.1 Build All Targets

Once configured, build the entire project:

cmake --build build --parallel

This compiles:

  • the instructor library (CourseLib)
  • the student library (StudentLib)
  • instructor examples (course/examples/*_target.cpp)
  • student examples (student/examples/*_target.cpp)
  • instructor unit tests (course/tests/*_unitTests.cc)
  • student unit tests (student/tests/*_unitTests.cc)

All dependencies are managed automatically by CMakeLists.txt.

1.2 Running Example Executables

The project generates example executables for both instructor and student code. These can be run manually for debugging or demonstration purposes.

Instructor example:

./build/LectureExample_target

Student example:

./build/StudentExample_target

While these binaries are helpful for understanding program flow, the primary grading workflow relies on CTest, not on executing examples manually.

1.3 Notes and Best Practices

  • Re-run cmake -B build only when configuration options change
    (e.g., enabling or disabling coverage).
  • Re-run cmake --build build after modifying any source file.
  • Use rm -rf build to fully reset the build environment.
  • Never build inside course/ or student/; this breaks portability.

Running Unit Tests

This page explains how unit tests are discovered, compiled, and executed using CTest.

1.0 Test Discovery

Any file named:

*_unitTests.cc

placed inside:

  • course/tests/
  • student/tests/

is automatically:

  • compiled into a dedicated test executable
  • registered as a CTest test

CTest discovers tests dynamically when running from the build/ directory.

1.1 Running All Tests

After building the project, run all tests with:

ctest --test-dir build --output-on-failure

This executes:

  • instructor tests, which validate student library code
  • student tests, which students may write for TDD (Test‑Driven Development)

CTest will display any failing assertions or crashes.

1.2 How Tests Link to the Libraries

Instructor tests typically link against StudentLib so they can validate student implementations.

Example CMake snippet:

target_link_libraries(${test_name}
PRIVATE
StudentLib
)

Student tests do the same, allowing students to verify their own code.

1.3 Best Practices

  • Run tests after every major change.
  • Commit your tests — they are part of the assignment.
  • Use small, focused test cases.
  • Prefer testing functions in isolation rather than only through example executables.

Coverage Testing

This page explains how to enable coverage flags, run tests, and generate coverage reports using gcovr.

1.0 Overview

Code coverage measures how much of your implementation
(course/src/ and student/src/) is executed by the unit tests.

The workflow is:

  1. Configure the project with coverage instrumentation enabled
  2. Build all targets
  3. Run the full unit test suite (CTest)
  4. Generate a coverage report using gcovr

Coverage works best on:

  • Linux (GCC) — fully compatible
  • macOS (Clang + llvm-cov) — requires LLVM tools installed

1.1 Enabling Coverage Flags

Coverage must be enabled during CMake configuration, before any targets are created. Otherwise, no object files will contain instrumentation.

Configure and enable coverage:

rm -rf build
cmake -B build \
-DCMAKE_BUILD_TYPE=Debug \
-DENABLE_COVERAGE=ON

Then build everything:

cmake --build build --parallel

‍Note:
On macOS you must install LLVM tools:

brew install llvm gcovr

</blockquote>

1.2 Running Tests (Generate Coverage Data)

Run all tests:

ctest --test-dir build --output-on-failure

This produces .gcno and .gcda files inside the build/ directory.
These files contain the execution counters used by gcovr.

1.3 Basic Coverage Report

From the repository root:

gcovr -r . --object-directory build

This prints a text report summarizing coverage per file.

1.4 Excluding Examples and Tests

Examples (*_target.cpp) and test files (*_unitTests.cc) usually show 0% coverage because they aren’t executed by other tests.

Exclude them to focus on the implementation:

gcovr -r . --object-directory build \
--exclude 'course/examples' \
--exclude 'student/examples' \
--exclude 'course/tests' \
--exclude 'student/tests'

Or exclude via patterns:

gcovr -r . --object-directory build \
--exclude '.*_target\.cpp' \
--exclude '.*_unitTests\.cc'

1.5 HTML Coverage Report

Generate a full HTML report:

mkdir -p coverage_html/
gcovr -r . --object-directory build \
--exclude 'course/examples' \
--exclude 'student/examples' \
--exclude 'course/tests' \
--exclude 'student/tests' \
--html-details coverage_html/coverage.html

Open:

coverage_html/coverage.html

‍Tip:
--html generates summary only.
--html-details generates line-by-line annotated coverage (recommended).

Documentation with Doxygen

This page explains how documentation is structured, configured, and generated using Doxygen as part of the Local Workflow.

1.0 Configuration

The main Doxygen configuration file is located at:

docs/Doxyfile

The entry point (home page) for the generated documentation is:

docs/mainpage.dox

Additional Course Guide documentation pages are stored in:

docs/CourseGuide/

Each .dox file in this directory contributes a section to the Course Guide.

1.1 Generating Documentation

From the repository root, run:

doxygen docs/Doxyfile

The generated HTML output will be placed in the directory specified by OUTPUT_DIRECTORY inside the Doxyfile, typically:

docs_out/html

Open the index.html file inside that directory to view the documentation.

1.2 Writing Doxygen Comments

Common Doxygen tags used in this project include:

  • @file — Describes the purpose of a file
  • @brief — One-line summary
  • @page — Standalone documentation page
  • @defgroup — Defines a documentation group (e.g., CourseGuide)
  • @ingroup — Assigns a page to a group
  • @code / @endcode — Code example blocks

Anything written in .dox files is interpreted as Markdown with Doxygen extensions.

1.3 Course Guide Structure

Groups for the Course Guide are defined in:

docs/CourseGuide/00_CourseGuide.dox

The available groups are:

  • CourseGuide — Main topic
  • CG_RepositorySetup
  • CG_DirectoryStructure
  • CG_LocalWorkflow
  • CG_GitHubWorkflow
  • CG_Lesson0

Every .dox page inside the Course Guide folder belongs to one of these groups.
